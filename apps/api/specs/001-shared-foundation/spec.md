## User Scenarios & Testing *(mandatory)*

### User Story 1 - 일관된 API 응답 수신 (Priority: P1)

클라이언트 개발자는 모든 API 엔드포인트에서 동일한 구조의 응답을 받는다. 성공 시에는 데이터와 성공 상태를, 실패 시에는 에러 코드와 메시지를 받는다.

**Why this priority**: 응답 구조가 통일되지 않으면 이후 모든 기능 단계에서 클라이언트가 각각 다른 방식으로 응답을 처리해야 하며, 가장 먼저 정의되어야 나머지 기반이 그 위에 쌓일 수 있다.

**Independent Test**: 헬스체크 엔드포인트 하나만 만들어도 성공/실패 응답 구조를 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 서버가 정상 동작 중일 때, **When** 클라이언트가 유효한 요청을 보내면, **Then** `{ "success": true, "data": ... }` 형태의 응답을 받는다.
2. **Given** 클라이언트가 잘못된 요청을 보낼 때, **When** 서버가 이를 거부하면, **Then** `{ "success": false, "error": { "code": "...", "message": "..." } }` 형태의 응답을 받는다.
3. **Given** 요청 값 검증에 실패했을 때, **When** 서버가 응답을 반환하면, **Then** 어떤 필드가 왜 실패했는지 상세 정보가 포함된다.

---

### User Story 2 - 비즈니스 예외와 시스템 예외 구분 수신 (Priority: P2)

클라이언트 개발자는 서버 오류가 발생했을 때 예상 가능한 비즈니스 예외(잘못된 요청, 권한 없음 등)와 예상치 못한 시스템 오류를 구분해서 받는다.

**Why this priority**: 클라이언트가 재시도, 사용자 안내, 오류 보고 중 어떤 처리를 할지 결정하려면 예외 종류를 구분할 수 있어야 한다.

**Independent Test**: 의도적으로 비즈니스 예외를 발생시키는 테스트와 시스템 예외를 유발하는 시나리오로 각각 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 비즈니스 규칙 위반이 발생했을 때, **When** 서버가 응답을 반환하면, **Then** 명확한 에러 코드와 클라이언트가 이해할 수 있는 메시지가 포함된다.
2. **Given** 예상치 못한 서버 내부 오류가 발생했을 때, **When** 서버가 응답을 반환하면, **Then** 내부 구현 정보(스택 트레이스, DB 쿼리 등)가 노출되지 않는 공통 오류 응답을 받는다.

---

### User Story 3 - Public / Admin API 접근 경계 확인 (Priority: P3)

클라이언트 개발자는 Public 엔드포인트는 인증 없이 접근할 수 있고, Admin 엔드포인트는 인증이 필요한 영역으로 분리되어 있음을 확인할 수 있다.

**Why this priority**: 실제 인증 구현은 이후 단계에서 하지만, 경계가 지금 정의되어 있어야 이후 Admin 기능이 올바른 위치에 배치된다.

**Independent Test**: `/api/public/` 경로는 인증 없이 200을 반환하고, `/api/admin/` 경로는 인증 없이 401을 반환하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 인증 정보가 없을 때, **When** `/api/public/` 하위 엔드포인트에 요청하면, **Then** 정상 응답(2xx)을 받는다.
2. **Given** 인증 정보가 없을 때, **When** `/api/admin/` 하위 엔드포인트에 요청하면, **Then** 인증 필요 응답(401)을 받는다.

---

### User Story 4 - 페이지네이션 응답 구조 확인 (Priority: P3)

클라이언트 개발자는 목록 조회 API에서 데이터와 함께 현재 페이지, 페이지 크기, 전체 개수, 다음 페이지 존재 여부를 일관된 구조로 받는다.

**Why this priority**: 목록 API가 생기기 전에 구조를 미리 정의해두어야 이후 작업물/카테고리 목록 단계에서 재정의 없이 재사용할 수 있다.

**Independent Test**: 페이지네이션 구조를 반환하는 샘플 엔드포인트로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 목록 조회 요청에 페이지 파라미터가 포함될 때, **When** 서버가 응답하면, **Then** `{ "content": [...], "page": N, "size": N, "totalElements": N, "hasNext": bool }` 구조를 포함한다.

---

### Edge Cases

- 요청 본문이 완전히 비어있을 때 검증 오류가 적절히 반환되는가?
- 페이지 번호가 전체 페이지 수를 초과할 때 빈 목록이 반환되는가?
- Admin 경로에 인증 헤더가 없을 때와 잘못된 값일 때 동일하게 401이 반환되는가?
- 시스템 예외 발생 시 응답에 스택 트레이스가 포함되지 않는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 서버는 모든 성공 응답에 대해 `success: true`와 결과 데이터를 포함한 일관된 응답 구조를 반환해야 한다.
- **FR-002**: 서버는 모든 실패 응답에 대해 `success: false`, 에러 코드, 에러 메시지를 포함한 일관된 응답 구조를 반환해야 한다.
- **FR-003**: 서버는 요청 값 검증 실패 시 어떤 필드가 왜 실패했는지 상세 정보를 에러 응답에 포함해야 한다.
- **FR-004**: 서버는 비즈니스 예외와 시스템 예외를 구분해서 처리해야 한다.
- **FR-005**: 서버는 시스템 예외 발생 시 내부 구현 정보를 클라이언트에게 노출하지 않아야 한다.
- **FR-006**: 서버는 공통 에러 코드를 열거형으로 관리해야 한다.
- **FR-007**: 서버는 도메인 데이터의 생성 시각과 수정 시각을 자동으로 관리할 수 있는 공통 기반을 제공해야 한다.
- **FR-008**: 서버는 목록 조회 응답에서 데이터, 현재 페이지, 페이지 크기, 전체 개수, 다음 페이지 존재 여부를 포함한 일관된 구조를 반환해야 한다.
- **FR-009**: `/api/public/` 경로는 인증 없이 접근 가능해야 한다.
- **FR-010**: `/api/admin/` 경로는 인증이 없으면 401 응답을 반환해야 한다.
- **FR-011**: shared 영역은 work, category, media, chat, ai 등 특정 비즈니스 도메인에 의존하지 않아야 한다.

### Key Entities

- **ApiResponse**: 단일 항목 성공 응답 — 성공 여부, 결과 데이터를 포함
- **ErrorResponse**: 실패 응답 — 성공 여부, 에러 코드, 에러 메시지, 필드별 상세 오류 포함
- **ErrorCode**: 공통 에러 코드 열거형 — 비즈니스 예외와 시스템 예외를 구분하는 코드 집합
- **PageResponse**: 페이지네이션 응답 — 데이터 목록, 현재 페이지, 페이지 크기, 전체 개수, 다음 페이지 여부 포함
- **BaseTimeEntity**: 공통 시간 엔티티 — 생성 시각(자동), 수정 시각(자동) 포함

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 모든 API 엔드포인트가 성공/실패 여부와 관계없이 동일한 최상위 응답 구조를 반환한다.
- **SC-002**: 비즈니스 예외 발생 시 지정된 에러 코드와 메시지가 정확히 반환되며, 내부 구현 정보는 포함되지 않는다.
- **SC-003**: 요청 값 검증 실패 시 실패한 필드 목록과 각 실패 이유를 응답에서 확인할 수 있다.
- **SC-004**: Public 경로는 인증 없이 100% 접근 가능하고, Admin 경로는 인증 없이 100% 차단된다.
- **SC-005**: 목록 조회 응답에서 항상 페이지 메타 정보(페이지 번호, 크기, 전체 수, 다음 페이지 여부)를 확인할 수 있다.
- **SC-006**: shared 패키지가 특정 비즈니스 도메인 패키지를 단 하나도 import하지 않는다.

## Assumptions

- 클라이언트는 REST API를 통해 JSON 형식으로 통신한다.
- 실제 관리자 인증(JWT, 세션 등) 구현은 이번 단계 범위 밖이며, Admin 경계만 준비한다.
- 페이지네이션은 오프셋 기반(page/size)을 기본으로 한다.
- 에러 코드는 문자열 상수로 관리하며, 이후 도메인별 에러 코드가 추가될 수 있다.
- 시간 필드는 UTC 기준으로 저장한다.
